Create `generate-declension.js` as an ESM Node.js script following the Phase 7 `generate-comparison.js` pattern exactly (import from 'fs', embedded tables, processBank dual-write, PROJECT_ROOT from cwd).

**1. ENDINGS tables (constant — same for all adjectives):**
Define three ending tables: stark, schwach, gemischt. Each has 4 cases (nominativ, akkusativ, dativ, genitiv) × 4 genders (maskulin, feminin, neutrum, plural). Use the exact values from 08-RESEARCH.md Pattern 1.

**2. POSITIV_STEM_EXCEPTIONS table:**
Only 4 entries where the positiv declension stem differs from the base word:
- `hoch_adj` → `hoh` (hoher, hohe, hohes, hohem — NOT hochem)
- `dunkel_adj` → `dunkl` (dunkler, dunkle, dunkles, dunklem)
- `flexibel_adj` → `flexibl` (flexibler, flexible, flexibles)
- `teuer_adj` → `teur` (teurer, teure, teures, teurem)

Do NOT add parallel, sauber, bitter, lecker, or any -ell adjective to this table. They use the full word as-is.

**3. KOMPARATIV_EXCEPTIONS table:**
Handle `viel_adj` whose komparativ `mehr` does not end in `-er`. Provide hand-curated forms:
- `mehr` declined: stark (mehrerer/mehrere/mehreres/mehrere, mehreren/mehrere/mehreres/mehrere, mehrerem/mehrerer/mehrerem/mehreren, mehreren/mehrerer/mehreren/mehrerer), schwach (mehrere/mehrere/mehrere/mehreren, mehreren/mehrere/mehrere/mehreren, mehreren/mehreren/mehreren/mehreren, mehreren/mehreren/mehreren/mehreren), gemischt (mehrerer/mehrere/mehreres/mehreren, mehreren/mehrere/mehreres/mehreren, mehreren/mehreren/mehreren/mehreren, mehreren/mehreren/mehreren/mehreren).
NOTE: `mehr` as a comparative adjective is grammatically unusual. Treat it the same as all others by applying standard endings to the stem `mehr` — producing `mehrerer`, `mehrere`, `mehreres`, etc. This is consistent with the algorithm. The special handling is just that `mehr` does not end in `-er` so the generic code path must detect this and still use `mehr` as the full base for `makeArticleBlock()`.

ACTUALLY — simpler approach per research: use `mehr` directly as the base in `makeArticleBlock('mehr')`. The algorithm `entry.comparison.komparativ` gives us `mehr`. Since we apply endings to the FULL komparativ form (not a stripped stem), `mehr` + `er` = `mehrerer` etc. This is correct. The only pitfall was if the code tried to strip `-er` from `mehr` first — but the algorithm does NOT strip. It uses the full komparativ form as the base. So `viel_adj` actually needs NO special exception — the standard algorithm produces correct forms. Verify this in Task 2.

**4. Block builder functions:**
- `makeCaseBlock(stem, endingsTable)` — applies endings to stem, returns {nominativ: {maskulin, feminin, neutrum, plural}, ...}
- `makeArticleBlock(stem)` — returns {stark, schwach, gemischt} each from makeCaseBlock
- `makeSuperlativBlock(supStem)` — returns {schwach: makeCaseBlock(supStem, ENDINGS.schwach)}

**5. buildDeclension(id, entry) function:**
- If `entry.undeclinable` → return null (skip)
- Get positiv stem: `POSITIV_STEM_EXCEPTIONS[id] ?? entry.word`
- Build `declension.positiv = makeArticleBlock(positivStem)` (all 360 declinable get this)
- If NOT nicht_komparierbar AND has comparison.komparativ:
  - `declension.komparativ = makeArticleBlock(entry.comparison.komparativ)` — use FULL komparativ form as base (schneller → schnellerer, höher → höherer, besser → besserer, mehr → mehrerer)
  - `declension.superlativ = makeSuperlativBlock(entry.comparison.superlativ)` — apply schwach endings to superlativ stem (schnellst → schnellste, best → beste, höchst → höchste)
- Return declension object

**6. DECLENSION_ALTERNATIVES table:**
At the entry level (NOT inside declension), store alternative accepted forms for adjectives with genuine Duden-recognized variants. The `adjectiveEntry` in the schema has no `additionalProperties` restriction, so a `declension_alternatives` key is allowed. The `declension` $def has `additionalProperties: false`, so alternatives CANNOT go inside it.

Structure: sparse object — only cells that have genuine Duden alternatives. Focus on `teuer_adj` (teur-/teuer- variants) as the primary confirmed case. For other `-el` and `-er` adjectives in the exception table, the Duden-preferred form is already the primary — only add alternatives where both forms are genuinely accepted by Duden.

Known alternatives for `teuer_adj`:
- The primary form uses stem `teur-` (teures, teurem, teurer, teure)
- Duden also accepts `teuer-` forms in many cells: teuerer, teueres, teuerem, teuere
- Store these as `declension_alternatives` at entry level

For each adjective in POSITIV_STEM_EXCEPTIONS, evaluate whether the non-exception form (using the full word) produces Duden-accepted alternatives. For `hoch_adj`: `hoch-` forms (hochem, hoches) are NOT accepted — no alternatives. For `dunkel_adj`: `dunkel-` forms (dunkelm, dunkeler) are NOT standard — the e-drop IS the standard. No alternatives. For `flexibel_adj`: similarly, e-drop IS the standard. For `teuer_adj`: both forms ARE accepted by Duden — add alternatives.

After applying `declension_alternatives` to applicable entries, log the count.

**7. processBank(filePath) function:**
Following Phase 7 pattern:
- Parse JSON
- Iterate all entries (skip `_metadata`)
- If `entry.undeclinable` → skip (these already have no declension key from Phase 7)
- Set `entry.declension = buildDeclension(id, entry)` — this handles BOTH cases: overwrites existing `declension: {}` and creates new key where missing (101 older entries)
- If entry has alternatives in DECLENSION_ALTERNATIVES, set `entry.declension_alternatives = DECLENSION_ALTERNATIVES[id]`
- Update `bank._metadata.generatedAt`
- Write JSON with `JSON.stringify(bank, null, 2) + '\n'`
- Log counts: populated, skipped (undeclinable), with_alternatives

**8. Main:**
