Create `generate-translations.js` in the phase directory following the established Phase 6/7/8 pattern (one-shot Node.js script with data embedded as constants).

**Script structure:**

1. **Read the core bank** (`vocabulary/core/de/adjectivebank.json`) to get the definitive list of 365 adjective _ids.

2. **Define all translation data as const objects.** For each of the 365 adjectives, provide:
   - `translation` (string): Primary meaning only — NO slashes
   - `explanation` (object): `{ "_description": "..." }` — explanation in target language. Norwegian explanations in Bokmål. English explanations in English.
   - `synonyms` (array of strings): Target-language synonyms for the PRIMARY meaning only
   - `examples` (array of `{ sentence, translation }` objects): German `sentence` is IDENTICAL between nb and en entries for the same adjective; only `translation` differs
   - `alternativeMeanings` (array of `{ meaning, context }` objects): ONLY when the German word has genuinely distinct secondary senses (not just synonyms). Omit this field entirely when not applicable.

3. **Structure the const objects for sentence sharing:**
   Define German example sentences once in a shared const (or define nb entries first, then reference the same `sentence` values for en). This ensures the locked decision "same German sentences in nb and en" is architecturally enforced, not just hoped for.

4. **Handle existing entry categories:**
   - 259 new entries: Full rich data for both nb and en
   - 24 simple nb entries (listed in RESEARCH.md): Upgrade to full rich format (add explanation, synonyms, examples)
   - 82 rich nb entries: Clean up `translation` field only if slash-separated (23 entries identified in RESEARCH.md). For slash cleanup: move primary meaning to `translation`, secondary synonyms to `synonyms`, genuinely different senses to `alternativeMeanings`. Preserve existing `explanation`, `synonyms`, and `examples` from the current file — do NOT regenerate these fields for entries that already have them.
   - 106 en entries: All currently simple (translation-only) — upgrade all to full rich format

5. **Merge logic** (same pattern as Phase 6 generate-stubs.js):
   - Read existing file → destructure `_metadata` from entries
   - For nb: merge new/upgraded entries over existing (new wins for upgrades/slash-cleanup); for entries NOT in the upgrade/new set, keep existing data unchanged
   - For en: merge all 365 entries (all need upgrade or are new)
   - Sort all non-metadata keys alphabetically (ASCII sort on _id strings — umlauts already transliterated)
   - Write with `_metadata` first, then sorted entries
   - Append trailing newline

6. **Update manifests:**
   - `vocabulary/translations/de-nb/manifest.json`: Set `adjectivebank.json` count to actual entry count (365). Update `totalWords` by adding 259 to current value.
   - `vocabulary/translations/de-en/manifest.json`: Same updates.
   - Note: Count only keys that don't start with `_` when computing entry count. Do NOT count `_metadata`.

7. **Run the script:** `node .planning/phases/09-translations/generate-translations.js`

**Translation quality guidelines:**
- Use Claude's linguistic knowledge for all translations. AI-generated is the locked decision.
- For false friends (arm=fattig not arm, fest=fast not fest, brav=snill not modig, rein=ren not rein, eventuell=muligens not eventuelt): extra care with explanation noting the false friend trap.
- For multi-meaning words: primary meaning in `translation`, genuinely distinct secondary senses in `alternativeMeanings` (see RESEARCH.md table for pre-identified candidates).
- Explanation `_description` format: Brief, useful description for language learners. Norwegian Bokmål for nb, English for en. Include usage notes, false friend warnings, or register notes where relevant.
- Example sentences: 1-3 per entry, Claude's discretion based on word complexity. Simple A1 words may need only 1 example; nuanced B1 words may benefit from 2-3 showing different contexts.

**CRITICAL format rules (from RESEARCH.md pitfall analysis):**
- `explanation` is ALWAYS `{ "_description": "..." }` — never a plain string
- `translation` is ALWAYS a single primary meaning — never slash-separated
- `synonyms` are ALWAYS target-language synonyms for the primary meaning — not mixed senses
- `alternativeMeanings` is ONLY present when genuinely distinct secondary senses exist — omit the field entirely otherwise
